01 · What this job actually is
Role Overview

Automation Controls Engineer — Manufacturing & Assembly

This is a manufacturing-floor engineering role at GM's Technical Center India (GMTCI) in Bangalore. You will not be writing web apps or backend services. Your job is to make physical machines on a factory floor behave correctly — and when they don't, figure out why.

Day-to-day work involves:

  • COREProgramming and maintaining PLCs that control assembly line machines
  • COREMonitoring SCADA/HMI dashboards for faults and production data
  • COREWorking on GPS (General Purpose Stations) and CNC equipment
  • COREParsing data from factory floor devices and surfacing it in reports
  • CORECoordinating with plant teams on project rollouts and maintenance

Why your software background helps:

  • EDGEPLC programming IS programming — logic, loops, states, conditions
  • EDGEPython for data parsing is something most controls engineers don't know
  • EDGEDashboarding with Power BI / Tableau is a differentiator in this field
  • EDGEVersion control, documentation habits, structured thinking
What GM expects at your level

You're not expected to be a professional. You ARE expected to show this:

✓ Must demonstrate

  • Conceptual clarity on what PLCs do and why
  • You've actually written ladder logic, even simulated
  • Understanding of automotive manufacturing context
  • Python competency — pandas, data parsing, visualisation
  • Basic Power BI or Tableau dashboard you built
  • Passion — you learned this on your own time
  • Ability to communicate technical ideas clearly

✗ You will NOT be judged on

  • Years of hands-on PLC field experience
  • Knowing all of GM's proprietary standards
  • Deep CNC or robotics expertise
  • Having worked in a physical plant
  • Certifications from Rockwell or Siemens
  • Production line architecture expertise

Week 09 Build Your Capstone Project — Manufacturing Analytics Dashboard 9 hrs ▼

Project: "Smart Factory OEE Monitor"

  • Simulate PLC/sensor data with Python
  • Parse and clean the data with pandas
  • Calculate OEE, MTBF, MTTR automatically
  • Build a live Streamlit or Power BI dashboard
  • Add anomaly detection (simple threshold alert)
  • Push code to GitHub with a clean README

What this project proves to GM

  • You understand manufacturing KPIs, not just code
  • You can bridge PLC data → actionable insight
  • Python + data analysis applied to their domain
  • Dashboard skills with Power BI or Tableau
  • You took initiative — no one asked you to do this
  • Software dev instincts: Git, README, clean code

HOW TO BUILD IT (STEP BY STEP)

  • STEP 1Generate synthetic manufacturing data: machine_id, timestamp, cycle_time, status, defect_flag (Python random/numpy)
  • STEP 2Write pandas code to compute: availability %, performance %, quality %, OEE score
  • STEP 3Add downtime categorization: planned, unplanned, changeover, breakdown
  • STEP 4Build Streamlit dashboard with: OEE gauge, downtime pie chart, trend line, anomaly alerts
  • STEP 5Import data into Power BI — build a second dashboard there for the "tools" requirement
  • STEP 6Push to GitHub. Write a README: "What it does, Why I built it, Tech used, How to run it"

05 · The capstone project — your proof
Project: Smart Factory OEE Monitor

This single project answers 6 of the 8 job requirements

Phase 1 — Simulate factory floor data (Python)

Generate synthetic PLC-like data: machine ID, timestamp, cycle time, fault codes, status flags. Represents requirement #7 (data parsing from manufacturing processes).

Phase 2 — Parse and analyse (pandas + numpy)

Calculate OEE, MTBF, MTTR. Classify downtime. Detect anomalies with simple threshold logic. Represents requirements #6 and #7.

Phase 3 — Live dashboard (Streamlit + Power BI)

Streamlit app with real-time charts. Import same data to Power BI for a second, more polished dashboard. Represents requirement #6 (Tableau, Power BI, Python).

Phase 4 — GitHub + documentation

Clean README with architecture diagram, setup steps, screenshots, and a written paragraph on what insights the data revealed. Shows communication + reporting skills (requirement #5).

Phase 5 — Extend (optional)

Add a simulated Modbus connection (pymodbus) that "polls" fake machine data every 5 seconds. This gets you closest to actual plant floor integration and is a massive talking point in interviews.
